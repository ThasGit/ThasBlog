springContext

Posted on 2019-04-01 | In 源码解读

关键接口

Beanfactory

org.springframework.beans.factory.Beanfactory负责bean的提取，包含getBean的一系列重载方法。这个接口只负责输出bean。

Object getBean(String name) throws BeansException;

ApplicationContext

ApplicationContext集成了多个接口，合成了一个管理应用上下文的接口，其继承结构如下图：

  • ResourcePatternResolver用于解析资源路径的，比如classpath*:这种路径就是该接口负责解析。
  • MessageSource用于读取配置文件信息，并且支持国际化。
  • ApplicationEventPublisher事件发布器。将会将ApplicationEvent事件交给注解了容器中注解了EventListener的方法去处理。
  • EnvironmentCapable用于获得应用上下文的环境，通常用于提供给那些接受了BeanFactory的方法，检查该工厂是否与自身匹配。

ConfigurableApplicationContext

ConfigurableApplicationContext继承了ApplicationContext和Lifecycle、Closeable，提供了对应用上下文声明周期的管理：

//Lifecycle
void start();
void stop();
boolean isRunning();
//Closeable
public void close() throws IOException;

ConfigurableApplicationContext内部定义了设置环境、设置父应用上下文、注册后处理器、注册协议处理器、事件发布、注册事件监听器、注册应用关闭回调方法、获取BeanFactory等方法。

ConfigurableListableBeanFactory

其实现DefaultListableBeanFactory是真正的beanFactory，最早实现是在GenericApplicationContext类和AbstractRefreshableApplicationContext(常用)中。ApplicationContext所有关于beanFactory的操作都是用的它，并且屏蔽了设置操作(Configurable)，ApplicationContext只能使用ListableBeanFactory提供的get操作。

AbstractRefreshableApplicationContext中的refreshable指BeanFactory能够被销毁并重新初始化，EmbeddedWeb这种内置了Web容器的是不能refresh的。对应的是用xml配置文件去初始化的ApplicationContext，而像使用注解的都是继承GenericApplicationContext不带refreshable的。

不管是哪种ApplicationContext，它们都是有refresh方法的，区别是能不能refreshBeanFactory()：

    protected final void refreshBeanFactory() throws IllegalStateException {
        if (!this.refreshed.compareAndSet(false, true)) {
        //  如果已经refresh过了 再次执行将会抛出异常
            throw new IllegalStateException(
                    "GenericApplicationContext does not support multiple refresh attempts: just call 'refresh' once");
        }
        this.beanFactory.setSerializationId(getId());
    }

初始化过程

ClassPathXmlApplicationContext

以ClassPathXmlApplicationContext为例：

  1. 构造方法中传入xml配置文件的路径。
  2. 构造过程会调用setConfigLocations方法，调用ConfigurableEvnironment中的解析器去解析提供的路径（解析其中的SpEl表达式，获得最终需要解析的路径。）如果环境不存在，将会立即创建一个StandardEnvironment。
  3. 设置完成之后，立即调用refresh()方法，根据配置文件刷新应用上下文。如果修改了spring的xml配置文件，是可以通过refresh方法重新刷新生效的。

refresh方法就是容器初始化的核心。

  1. prepareRefresh(beanFactory)
    主要工作是initPropertySources()。PropertySources就是想application.propeties和其他*.propeties这类定义属性值的配置文件。然后valid一下，校验一些必须要配置的属性。

    对于AnnotationConfig这种根据注解去初始化容器的还需要先执行this.scanner.clearCache()，清理ClassPathBeanDefinitionScanner的缓存。

  2. ConfigurableListableBeanFactory beanFactory = obtainFreshBeanFactory();
    硬翻译就是获得一个新鲜的bean工厂。这个时候就需要调用refreshBeanFactory()方法了。对于可refresh的容器，源码是这样的，不能refresh的这段是抛出异常。

     if (hasBeanFactory()) {
         //销毁原先创建的
         destroyBeans();
         closeBeanFactory();
     }
     try {
         DefaultListableBeanFactory beanFactory = createBeanFactory();
         beanFactory.setSerializationId(getId());
         customizeBeanFactory(beanFactory);
         //装载bean定义
         loadBeanDefinitions(beanFactory);
         synchronized (this.beanFactoryMonitor) {
             this.beanFactory = beanFactory;
         }
     }
     catch (IOException ex) {
         throw new ApplicationContextException("I/O error parsing bean definition source for " + getDisplayName(), ex);
     }

  3. prepareBeanFactory(beanFactory);
    主要是注入内置的bean。

  4. postProcessBeanFactory(beanFactory);
    空的

  5. invokeBeanFactoryPostProcessors(beanFactory)
    执行beanFactory的后处理器。这个必须手动add。这个后置处理器(BeanDefinitionRegistryPostProcessor)可用来手动添加一下beanDefination，比如使用注解注入方式的ApplicationContext就是用ConfigurationClassPostProcessor来注册beanDefination的。

  6. registerBeanPostProcessors(beanFactory);
    注册bean的后处理器。这个会在finishBeanFactoryInitialization中创建bean时使用。在创建过程中会被内置的BeanPostProcessor处理，等全部创建完成再注册到容器。所有自行创建的BeanPostProcessor不会相互影响。

  7. initMessageSource()
    MessageSource是用来支持国际化的。需要定义MessageSource的bean这一步才有用，springboot需要创建message-*.properties的文件。

  8. initApplicationEventMulticaster()
    初始化(如果定义了话)/注册(定义一个Simple多播器)事件多播器。多播器就是指当接收到事件时，将事件通知给所有监听该事件的监听器。

  9. onRefresh();
    第一步UiApplicationContextUtils.initThemeSource(this) 有点类似MessageSource，有选择性的加载属性值，这个一般用在WebMVC的网页主题。
    对于EmbeddedWeb内置web容器类型的，还需要创建web容器，createEmbeddedServletContainer()，默认是Tomcat。

  10. registerListeners();
    注册事件监听器。
    getApplicationEventMulticaster().addApplicationListener(listener)刚刚的事件多播器有用了。隔了一个onRefresh()，有可能web容器也需要注册事件。
    springApplicationEvent参考

  11. finishBeanFactoryInitialization(beanFactory)
    结束beanFactory的初始化过程/预创建单例bean。

    
    // 冻结bean定义 不期望这些定义被改变
    beanFactory.freezeConfiguration();

    // 初始化所有非懒加载的单例bean
    beanFactory.preInstantiateSingletons();

    Bean的生命周期，实例化->BeanNameAware->BeanFactoryAware->BeanPostProcessor前置方法->(内置Processor执行ApplicaiontContextAware)->InitializingBean->BeanPostProcessor后置方法。DisposableBean销毁方法将ContextStoppedEvent事件时在执行。初始化过程如下：

    protected Object initializeBean(final String beanName, final Object bean, RootBeanDefinition mbd) {
    if (System.getSecurityManager() != null) {
        AccessController.doPrivileged(new PrivilegedAction<Object>() {
            @Override
            public Object run() {
                invokeAwareMethods(beanName, bean);
                return null;
            }
        }, getAccessControlContext());
    }
    else {
        invokeAwareMethods(beanName, bean);
    }

    Object wrappedBean = bean;
    if (mbd == null || !mbd.isSynthetic()) {
        wrappedBean = applyBeanPostProcessorsBeforeInitialization(wrappedBean, beanName);
    }

    try {
        invokeInitMethods(beanName, wrappedBean, mbd);
    }
    catch (Throwable ex) {
        throw new BeanCreationException(
                (mbd != null ? mbd.getResourceDescription() : null),
                beanName, "Invocation of init method failed", ex);
    }
    if (mbd == null || !mbd.isSynthetic()) {
        wrappedBean = applyBeanPostProcessorsAfterInitialization(wrappedBean, beanName);
    }
    return wrappedBean;
}
private void invokeAwareMethods(final String beanName, final Object bean) {
    if (bean instanceof Aware) {
        if (bean instanceof BeanNameAware) {
            ((BeanNameAware) bean).setBeanName(beanName);
        }
        if (bean instanceof BeanClassLoaderAware) {
            ((BeanClassLoaderAware) bean).setBeanClassLoader(getBeanClassLoader());
        }
        if (bean instanceof BeanFactoryAware) {
            ((BeanFactoryAware) bean).setBeanFactory(AbstractAutowireCapableBeanFactory.this);
        }
    }
}
// 由内置的的ApplicationContextAwareProcessor执行
private void invokeAwareInterfaces(Object bean) {
    if (bean instanceof Aware) {
        if (bean instanceof EnvironmentAware) {
            ((EnvironmentAware) bean).setEnvironment(this.applicationContext.getEnvironment());
        }
        if (bean instanceof EmbeddedValueResolverAware) {
            ((EmbeddedValueResolverAware) bean).setEmbeddedValueResolver(this.embeddedValueResolver);
        }
        if (bean instanceof ResourceLoaderAware) {
            ((ResourceLoaderAware) bean).setResourceLoader(this.applicationContext);
        }
        if (bean instanceof ApplicationEventPublisherAware) {
            ((ApplicationEventPublisherAware) bean).setApplicationEventPublisher(this.applicationContext);
        }
        if (bean instanceof MessageSourceAware) {
            ((MessageSourceAware) bean).setMessageSource(this.applicationContext);
        }
        if (bean instanceof ApplicationContextAware) {
            ((ApplicationContextAware) bean).setApplicationContext(this.applicationContext);
        }
    }
}

  12. finishRefresh()
    结束ApplicationContext的刷新过程

AnnotationConfigApplicationContext

AnnotationConfigApplicationContext与xml方式有差别，绑定beanDefinition的操作不在obtainFreshBeanFactory()中进行load。注解扫描范围需要有Configuration来指定ComponentScan。AnnotationConfigApplicationContext将注解方式的bean的信息读取工作交给了做成了工厂后处理器ConfigurationClassPostProcessor，并由ConfigurationClassParser来解析。在执行invokeBeanFactoryPostProcessors(beanFactory)，对注解进行扫描。原型模式将不会有这样的生命周期，只负责实例化并交给用户。

再来说说bean的实例化过程。

bean的作用域可以有singleton、prototype两种，只有单例模式的bean在ApplicationContext创建过程中去创建的。创建bean的过程会调用getBean方法，bean的实例化放在了getBean方法中。如果一个bean的构造方法参数依赖了另一个bean，那么会先暂停这个bean的创建过程，转而去执行另一个bean的创建过程。如果只是属性依赖，那么将会先完成当前bean的实例化过程，再去创建依赖的bean，然后注入给属性。因此构造方法参数循环依赖，将会造成栈溢出的问题，因为双方都没有实例化完成，会不断的去尝试创建彼此的实例。

WebApplicationContext/ConfigurableWebApplicationContext

WebApplicationContext继承自ApplicationContext，加入了request、session作用域（最新版本源码没有看到global session），并且支持挂载和获取ServletContext、ServletConfig。

ConfigurableWebApplicationContext继承了原先的ConfigurableApplicationContext和WebApplicationContext，加入了WebApplicationContext的配置相关的操作。
